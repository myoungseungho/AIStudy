TOPIC 09

검증 데이터는 왜 또 필요해?

Q. 훈련/테스트 2개로 부족해?

Part 1. 테스트는 단 한 번만 써야 해

훈련/테스트 2개면 충분한 거 아니야? 검증(Validation) 데이터는 왜 필요해?

테스트 데이터는 "최종 시험"이라서, 미리 보면 안 돼.

비유로 설명해줄게.

비유: 수능 준비 모의고사 (검증 데이터): - 여러 번 봐도 OK - 점수 보고 공부 방법 조정 - "이 부분이 약하네" → 보완 수능 (테스트 데이터): - 딱 한 번만 - 이걸로 최종 평가 - 미리 보면 의미 없음 문제 상황: 모의고사 없이 수능만 있으면? → 어떻게 준비할지 모름 → 실력 확인 방법이 없음 모의고사를 수능으로 착각하면? → 모의고사 문제만 달달 외움 → 진짜 수능에서 망함

이해됐어?

검증은 "중간 점검", 테스트는 "최종 평가"야.

Part 2. 3가지 데이터 세트

그럼 데이터를 3개로 나누는 거야?

그래. 보통 이렇게 나눠.

데이터 분할 (1000개 기준): 1. 훈련 데이터 (Training Set): 600개 (60%) → AI가 패턴을 배우는 데이터 → "공부하는 교과서" 2. 검증 데이터 (Validation Set): 200개 (20%) → 학습 중간에 성능 확인 → "모의고사" → 여러 번 봐도 됨 3. 테스트 데이터 (Test Set): 200개 (20%) → 최종 성능 평가 → "수능" → 딱 한 번만! 비율은 상황에 따라 다름: - 6:2:2 (일반적) - 7:2:1 (데이터 많을 때) - 8:1:1 (데이터 적을 때)

3개로 나누면 각자 역할이 명확해져.

Part 3. 검증 데이터의 역할

검증 데이터로 구체적으로 뭘 하는데?

여러 가지 의사결정에 사용해.

검증 데이터 활용: 1. 조기 종료 (Early Stopping) 에폭 1: 검증 성능 80% 에폭 2: 검증 성능 85% 에폭 3: 검증 성능 88% 에폭 4: 검증 성능 87% ← 떨어지기 시작! → 에폭 3에서 멈춤 (과적합 방지) 2. 하이퍼파라미터 튜닝 학습률 0.001 → 검증 성능 82% 학습률 0.01 → 검증 성능 87% ← 더 좋네! 학습률 0.1 → 검증 성능 75% → 0.01 선택 3. 모델 선택 모델A: 검증 성능 83% 모델B: 검증 성능 89% ← 이거 선택! 모델C: 검증 성능 81% 4. 과적합 감지 훈련 성능: 95% 검증 성능: 70% → 과적합 발견! → 조치 필요

검증 데이터는 "개발 중 의사결정"에 써.

테스트는 절대 건드리지 말고!

Part 4. 교차 검증

검증 데이터가 너무 적으면 어떡해?

교차 검증(Cross-Validation)을 써.

K-Fold 교차 검증 (K=5 예시): 전체 데이터를 5개로 나눔: [1] [2] [3] [4] [5] Round 1: [2][3][4][5] 훈련, [1] 검증 → 성능 85% Round 2: [1][3][4][5] 훈련, [2] 검증 → 성능 87% Round 3: [1][2][4][5] 훈련, [3] 검증 → 성능 84% Round 4: [1][2][3][5] 훈련, [4] 검증 → 성능 86% Round 5: [1][2][3][4] 훈련, [5] 검증 → 성능 88% 최종 성능: (85+87+84+86+88)/5 = 86% 장점: - 모든 데이터가 검증에 한 번씩 사용됨 - 평균 성능이 더 신뢰도 높음 - 데이터 적을 때 유용 단점: - 5번 학습해야 해서 시간 오래 걸림

데이터가 부족하면 교차 검증이 필수야.

Part 5. 테스트는 정말 한 번만

테스트를 여러 번 보면 안 돼?

절대 안 돼! 테스트를 여러 번 보면 의미가 없어져.

잘못된 프로세스: 시도1: 모델A → 테스트 성능 80% "아, 별로네" 시도2: 모델B → 테스트 성능 85% "더 좋네!" 시도3: 모델C → 테스트 성능 90% "이거다!" 문제: → 테스트 성능을 보고 모델을 조정함 → 결국 테스트 데이터에 맞춰 튜닝한 셈 → "시험 문제 보고 공부"한 거랑 같음 → 일반화 성능 알 수 없음! 올바른 프로세스: 개발 단계 (검증 데이터 사용): 모델A → 검증 성능 80% 모델B → 검증 성능 85% 모델C → 검증 성능 90% ← 선택 최종 평가 (테스트 데이터, 딱 한 번): 모델C → 테스트 성능 88% → 이게 진짜 성능!

검증으로 개발하고, 테스트로 최종 확인만 해.

Part 6. 실전 workflow

그래서 실제로는 어떻게 진행해?

전체 흐름을 보여줄게.

AI 개발 전체 흐름: 1. 데이터 분할 전체 1000개 → 훈련 600 / 검증 200 / 테스트 200 (테스트는 금고에 보관하듯 숨겨둠) 2. 개발 & 튜닝 (검증 사용) - 여러 모델 시도 - 하이퍼파라미터 조정 - 검증 성능 보며 개선 → 수십, 수백 번 반복 OK 3. 최종 모델 선택 검증 성능이 가장 좋은 모델 선택 4. 최종 평가 (테스트 사용, 딱 한 번!) 테스트 데이터로 성능 측정 → 이 숫자가 공식 성능 5. 배포 실전에 투입 → 실제 사용자 데이터로 모니터링 6. 재학습 (필요시) 성능 떨어지면 새 데이터 모아서 재학습 → 새로 훈련/검증/테스트 분할

검증은 개발 도구, 테스트는 성적표야.

역할을 혼동하면 안 돼!

핵심 정리

  • 3개 세트 필요 - 훈련(학습), 검증(개발), 테스트(최종평가)
  • 검증은 여러 번 - 모델 선택, 하이퍼파라미터 튜닝에 사용
  • 테스트는 단 한 번 - 최종 성능 측정용, 절대 개발에 사용 금지
  • 교차 검증 - 데이터 적을 때 검증 데이터를 효율적으로 사용
  • 비율 - 보통 6:2:2 또는 7:2:1로 분할
  • 역할 구분 - 검증은 개발 도구, 테스트는 성적표
